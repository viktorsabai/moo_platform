# Диагностика: почему API «не работает» и «ничего не меняется»

## Общая картина

ЛК и приложение для клиентов — **два разных потока данных**, которые должны сходиться через `restaurantId`:

```
┌─────────────────┐                    ┌─────────────────┐
│   ЛК (admin)    │                    │  Клиент (/)     │
│                 │                    │                 │
│  ufo_restaurant │◄─── cookie ───────►│  ufo_restaurant │
│  cookie         │     + session      │  + session      │
└────────┬────────┘                    └────────┬────────┘
         │                                      │
         ▼                                      ▼
   getRestaurantContext()            getConsumerRestaurantId()
         │                                      │
         └──────────► restaurantId ◄────────────┘
```

**Проблема была в том, что `restaurantId` для клиента не всегда совпадал с тем, что выбирает владелец.**

---

## Что было не так

### 1. Связка «бот → заведение» для клиента

**Было:** При открытии по ссылке `t.me/bot?startapp=MOO` клиент мог получить `restaurantId = 'default'`, а не MOO.

**Почему:** `restaurantId` брался только из сессии (после логина) или из cookie. Cookie `ufo_restaurant` не выставлялась при входе по ссылке бота, пока пользователь не залогинится.

**Исправлено:** Добавлен API `/api/venue/init`, который:
- принимает `initData` из Telegram WebApp;
- верифицирует подпись;
- по `start_param` находит заведение и выставляет cookie `ufo_restaurant`.

`VenueProvider` теперь перед загрузкой контекста вызывает `/api/venue/init`, если есть `initData`, чтобы cookie устанавливалась до любых запросов к `/api/venue/context`.

### 2. Cookie не всегда уходили с запросами

**Было:** В части admin-запросов `fetch` не использовал `credentials: 'include'`, из-за чего в некоторых средах (WebView, реддиректы) cookie могли не отправляться.

**Исправлено:** Во всех admin-запросах добавлен `credentials: 'include'`.

### 3. Ошибки сохранялись без явной обратной связи

**Было:** `AdminQuickSettings` при ошибке PATCH тихо возвращал состояние, без сообщения.

**Исправлено:** При ошибке вызывается `toast.error(data?.error || 'Не удалось сохранить')`.

## Как связана логика

| Компонент            | Источник restaurantId                        | Когда обновляется                        |
|----------------------|----------------------------------------------|------------------------------------------|
| Admin layout         | `ufo_restaurant` cookie                      | При переключении заведения (switch API)  |
| Admin API (settings, orders и т.д.) | `getRestaurantContext()` → session + cookie | Каждый запрос                            |
| Consumer API (banners, menu и т.д.) | `getConsumerRestaurantId()` → session, cookie, default | Каждый запрос                            |
| VenueProvider        | `/api/venue/context`                         | При загрузке и смене страницы/видимости  |

**Важно:** Для корректной работы нужны:
1. **Бот и BotIntegration** — у заведения должен быть `startParam` (например, slug) и `botUsername` в `BotIntegration`.
2. **BOT_TOKEN** — в `.env` должен быть задан глобальный токен или токен бота для проверки `initData`.

---

## «0 заказов» и «Главная пустая»

- **0 заказов:** Данные реальные: в БД для текущего заведения нет заказов. ЛК показывает факт.
- **Главная пустая:** Fallback-баннеры удалены. Сейчас показываются только баннеры из ЛК. Если в ЛК 0 баннеров — на главной их тоже нет.

---

## Чеклист для проверки

1. [ ] В ЛК заведение подключено: QR → username бота и `startParam` заданы.
2. [ ] `BOT_TOKEN` в `.env` совпадает с токеном этого бота (или глобальный токен корректен).
3. [ ] Клиент заходит по ссылке `t.me/YourBot?startapp=YourStartParam`.
4. [ ] После входа в ЛК и переключения заведения cookie `ufo_restaurant` установлена (проверить в DevTools → Application → Cookies).
