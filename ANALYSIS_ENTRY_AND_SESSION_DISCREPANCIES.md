# Анализ: расхождения пайплайна входа и сессий с эталонной моделью

**Задача:** зафиксировать несоответствия между текущей реализацией (Telegram-бот + mini app) и эталонной продуктовой моделью. Код не менялся.

---

## 1. Фактический пайплайн входа (как реализовано в коде)

### 1.1. Обработка /start (бот)

- **Файл:** `src/app/api/telegram/webhook/route.ts`
- При команде `/start` бот **всегда** задаёт вопрос: «Вы пришли как гость или как владелец заведения?»
- Две inline-кнопки:
  - **«Я гость»** → `web_app: buildWebAppUrl('/')` — открывает mini app на главной.
  - **«Я владелец»** → `web_app: buildWebAppUrl('/profile')` — открывает mini app на странице профиля.
- Роль не сохраняется в боте; выбор только задаёт стартовый URL mini app.

### 1.2. Определение роли

- **Роль** определяется на бэкенде по данным сессии NextAuth (JWT): `platformRole`, `memberRole`, членство в `RestaurantMember` (`getRestaurantContext`, `authOptions` в `src/lib/auth.ts`, `src/lib/restaurant-context.ts`).
- В UI роль используется так:
  - **Профиль:** `isOwner = memberRole === 'OWNER' || memberRole === 'ADMIN'` (`src/app/profile/page.tsx`).
  - **Хедер и навбар:** `isOwnerContext(pathname) = pathname.startsWith('/admin') || pathname === '/profile/owner'` (`HeaderCompact`, `BottomNavbar`).
- Отдельного «контекста guest/owner» в смысле явного переключения роли в UI нет: есть только факт пути (`/admin`, `/profile/owner`) и наличие прав в сессии.

### 1.3. Когда активируется owner-контекст

- **По URL:** при заходе на `/admin` или `/profile/owner` показывается owner-хедер и owner-навбар («назад в профиль»).
- **Вход в owner из приложения:**
  - В профиле при `isOwner` показывается блок «режим владельца» с кнопкой «Войти в кабинет» → `Link href="/admin"`.
  - Переход сразу на `/admin`, без промежуточного экрана «управление заведением / открыть как гость».
- **Страница** `src/app/profile/owner/page.tsx` только делает `redirect('/profile')`, отдельного owner entry-экрана нет.

### 1.4. Как открывается mini app

- Из бота по кнопкам после `/start`: либо `/`, либо `/profile` (см. выше).
- Из других мест (заказы, подписки и т.д.): `buildWebAppUrl(...)` с нужным путём.
- При открытии mini app с `initData` в провайдере выполняется **автоматический** `signIn('telegram', { initData, redirect: false, callbackUrl: '/' })` (`TelegramAutoLogin` в `src/components/providers.tsx`). Отдельного экрана входа при этом может не быть (если пользователь не попал на `/signin`).

### 1.5. Экраны sign-in / sign-up

- **Sign-in:** `src/app/(auth)/signin/page.tsx`
  - В mini app (есть `Telegram.WebApp`): заголовок «Вход», «Войдите через Telegram», вызов `signIn('telegram', { initData, redirect: false })`, при успехе `router.replace('/')`.
  - Вне mini app (desktop): редирект на `t.me`-ссылку; блок «Войти с паролем» → `/signin/password`; ссылка «Войти как админ» → `/admin/login`; ссылка «Регистрация» → `/signup`.
- **Sign-up:** `src/app/(auth)/signup/page.tsx`
  - Текст «Регистрация», «первый вход через Telegram», в mini app кнопка «Продолжить» → `signIn('telegram', { initData, redirect: false })`, вне mini app — редирект в Telegram и ссылка «Уже есть аккаунт? Войти» → `/signin`.
- **Вход с паролем (desktop):** `src/app/(auth)/signin/password/page.tsx` — логин (Telegram ID) + пароль, `signIn('owner', { login, password, redirect: false })`, при успехе `router.replace('/profile')`.
- **Admin login:** `src/app/admin/login/page.tsx` — логин/пароль, `signIn('admin', ...)`, при успехе `router.replace('/admin')`.

### 1.6. Кнопка «Выйти» и повторное открытие

- **«Выйти»:** `src/app/profile/page.tsx`: `signOut({ callbackUrl: '/signin' })`. Выполняется полноценный выход из NextAuth (сессия сбрасывается) и редирект на `/signin`.
- **Повторное открытие mini app:** при следующем открытии с тем же Telegram-контекстом снова срабатывает `TelegramAutoLogin`: при наличии `initData` и отсутствии сессии вызывается `signIn('telegram', ...)`. Пользователь снова оказывается «залогинен» без явного действия.

## 2. Расхождения с эталонным пайплайном входа (guest/owner)

| # | Проблема | Где в коде | Пункт эталона |
|---|----------|------------|----------------|
| E1 | **На /start всегда спрашивается роль** («гость или владелец»). Эталон: по умолчанию НЕ спрашивать роль, пользователь стартует как guest. | `src/app/api/telegram/webhook/route.ts` — текст приветствия и две кнопки «Я гость» / «Я владелец». | П.2: «По умолчанию НЕ спрашивать роль». |
| E2 | **«Я владелец» открывает сразу /profile**, а не owner entry. Нет экрана выбора «Перейти к управлению» / «Открыть как гость». | Webhook: `buildWebAppUrl('/profile')` для «Я владелец». Нет маршрута/экрана owner entry. | П.4–5: owner flow = вход в owner onboarding/entry; при наличии заведений — entry-экран с выбором. |
| E3 | **Нет проверки «есть ли у user_id заведения»** на входе в owner: не запускается явный owner onboarding при отсутствии заведений и не показывается entry-экран при их наличии. | Нет отдельного handler/страницы для owner entry после нажатия «Я владелец». | П.5: «Если у user_id НЕТ заведений → onboarding; ЕСТЬ → entry-экран». |
| E4 | **Вход в режим владельца из mini app без entry-экрана:** из профиля «Войти в кабинет» ведёт сразу на `/admin`, без экрана выбора заведения и без выбора «управление / открыть как гость». | `src/app/profile/page.tsx` — `Link href="/admin"` в блоке «режим владельца». | П.7: «сначала entry-экран владельца, выбор заведения (если несколько), только потом admin». |
| E5 | **Страница `/profile/owner` не реализует owner entry:** только редирект на `/profile`. | `src/app/profile/owner/page.tsx`: `redirect('/profile')`. | П.5, П.7: entry-экран владельца. |
| E6 | **Mini app при открытии по «Я владелец» попадает на /profile**, а не в guest-контекст с последующим явным входом в owner. Эталон: mini app всегда открывается в guest-контексте. | Webhook: «Я владелец» → `buildWebAppUrl('/profile')`. | П.6: «Mini app всегда открывается в guest-контексте». |

Эталон: нет экранов входа/регистрации, нет классического sign-in/sign-up. | `src/app/(auth)/signin/page.tsx`, `src/app/(auth)/signup/page.tsx`. | П.1 (нет sign-in/sign-up), П.5 (нет экранов входа при первом запуске). |

## 4. Краткая сводка

- **Пайплайн входа (guest/owner):** на `/start` принудительно спрашивается роль; кнопка «Я владелец» ведёт в `/profile` без owner entry и без проверки «есть ли заведения»; вход в кабинет из профиля — сразу `/admin`, без entry-экрана и выбора заведения; mini app при выборе «Я владелец» открывается на `/profile`, а не в guest-контексте.
- **Идентификация и сессии:** присутствуют экраны sign-in/sign-up; «Выйти» реализован как logout с переходом на `/signin`; при отсутствии сессии в профиле показывается «войти»; есть вход по паролю (owner и admin), что противоречит модели «только telegram_user_id, без логинов/паролей».

Документ фиксирует только расхождения; изменения в коде и UI не предлагались.
